# Stage 1 — стабилизация bot-creter

Рантайм-модель не менялась: боты по-прежнему поднимаются подпроцессами через `services/bot_runner.py`. Изначально Stage 1 писался под VDS (systemd), затем проект решили вернуть на **Railway** — деплой теперь через `git push` (авто-сборка Railway), напрямую в Anthropic API (`ANTHROPIC_API_KEY`, без OpenRouter). `deploy/bot-creter.service` от VDS-варианта оставлен в репозитории как есть — Railway его не использует и не читает, он не мешает.

## Раунд 3 — доводка Stage 1 (безопасные правки из ревью)

### Задача 1 — токен убран из логов
`services/telegram_api.py:18` логировал весь ответ `getManagedBotToken` (содержал токен нового бота) — теперь только `data.get('ok')`. Проверил весь проект на аналогичные утечки: `handlers/create_bot.py:411,428` логируют `managed_data` (метаданные Telegram-события — `bot_id`/`user_id`), не сам токен — реальной утечки там нет, токен там никогда не извлекается. Больше утечек не найдено.

### Задача 2 — warning на незаданный `OWNER_ID`
`main()` в `main.py` теперь пишет `logger.warning`, если `OWNER_ID` не задан или равен 0 — чтобы забытая переменная на Railway была сразу видна в логах, а не тихо блокировала управление ботами для всех.

### Задача 3 — порт `tour_operator.py`
Проверка показала: порт уже читается из env (`PORT = int(os.getenv("PORT", "8080"))`) — эта часть была сделана раньше, шаблон трогать не пришлось. Реальная причина потенциального конфликта — на уровне оркестрации (`services/bot_runner.py` не назначал разным ботам разные значения `PORT`). Тестовая правка (уникальный `PORT` на бота) была сделана и затем **откачена**: ревью обнаружило, что она меняет проблему «падает при коллизии» на «недоступно снаружи» — Railway пробрасывает публичный трафик только на один порт, назначенный сервису, а не на производный. Это глубже пары строк и требует отдельного архитектурного решения (например: у каждого веб-бота — свой Railway-сервис, либо общий reverse-proxy внутри фабрики). **Оставлено как открытый вопрос**, ничего не менял в проде.

### Задача 4 — лок от двойного клика на дорогих операциях
Два слоя защиты:
- `services/bot_runner.py`: `start_bot`/`stop_bot` теперь используют `asyncio.Lock` по `bot_id` (`_get_lock`), с приватным `_stop_bot_unlocked` для вызова изнутри уже захваченного лока (без дедлока) — это чинит саму неатомарность спавна процессов.

По находке ревью — тем же паттерном заодно закрыл `cb_start`/`cb_restart` (изначально не планировалось, но лок должен был явно покрывать и старт согласно тексту задачи).

### Задача 5 — дубли карточек при двойном клике
Добавлен хелпер `_edit_or_resend()`: ловит `TelegramBadRequest` и если это «message is not modified» — молча игнорирует (карточка уже актуальна), иначе шлёт новое сообщение как раньше. Применён в `cb_stop`, `cb_restart`, и (по находке ревью) в `cb_start`.

### Задача 6 — `PRAGMA journal_mode=WAL`
Добавлена первой строкой в `init_db()`. WAL — персистентное свойство файла БД (хранится в заголовке), устанавливается один раз при первом подключении, повторять в каждом `connect()` не нужно. Протестировано: `PRAGMA journal_mode` после `init_db()` возвращает `wal`.

**Операционная заметка от ревью:** WAL создаёт файлы `bots.db-wal`/`bots.db-shm` рядом с `bots.db`. Если когда-нибудь появится бэкап-скрипт — он должен копировать всю папку `DATA_DIR`, а не только `bots.db`, иначе последние закоммиченные, но не checkpoint'нутые записи потеряются при восстановлении. Сейчас отдельного бэкапа нет, это просто задел на будущее.

### Задача 7 — атомарность создания бота
`create_bot_record` удалена, вместо неё `create_bot_record_with_admins(name, description, token, file_path, admin_ids, username=None)` — INSERT бота и всех админов (через `INSERT OR IGNORE`, с дедупликацией) в одной транзакции. Оба места вызова в `handlers/create_bot.py` обновлены (список `admin_ids` собирается заранее: создатель + владелец, если отличается). Проверено: `aiosqlite` без явного commit откатывает всю транзакцию при ошибке — крах между вставками невозможен. Права доступа в `create_bot.py` не трогал — это прямой запрет ТЗ.

### Задача 2 — owner-гейт в `handlers/manage_bots.py` и `handlers/general.py`
Раньше `handlers/manage_bots.py` не проверял права вообще нигде — любой пользователь Telegram мог управлять чужими ботами. Добавлена проверка `_is_owner(user_id)` (импортируется из `handlers/admin_manager.py`, сравнивает с `OWNER_ID` из env) первой строкой в **каждый** из 16 хендлеров: команды `/list`, `/stop`, `/run`, `/logs`; коллбэки `list`, `info:`, `start:`, `stop:`, `restart:`, `logs:`, `delete:`, `recreate:`, `autofix:`, `fixbug:`; FSM-хендлеры `msg_fix_voice`, `msg_fix_text`. Для callback-хендлеров проверка стоит **до** `await callback.answer()`, чтобы не отвечать успехом до проверки прав. Не-владельцу — короткое сообщение-отказ (`_deny_message`/`_deny_callback`) и `return`.

То же самое для `handlers/general.py`: `general_voice`/`general_text` (catch-all, вызывают платный `ask_assistant`) теперь тихо игнорируют сообщения не от владельца.

**Дополнительный блокер, найденный ревью и исправленный сразу:** в 5 местах `manage_bots.py` (логи, ошибки автофикса/пересоздания) сырой текст (stderr упавшего бота, `str(e)`) вставлялся в HTML-сообщения без экранирования — traceback почти всегда содержит `<module>`, `<listcomp>` и т.п., что ломает парсинг HTML в Telegram (`TelegramBadRequest`) прямо в тот момент, когда владелец пытается посмотреть логи или диагностику упавшего бота. Обёрнуто в `html.escape()`.

### Задача 3 — миграция токенов по формату, а не по «не расшифровалось»
`migrate_encrypt_tokens()` в `db/database.py` переписана: раньше plaintext определялся по неудаче расшифровки текущим ключом — это путало «не зашифровано» с «зашифровано другим/старым ключом» и грозило **необратимой порчей** токенов при смене `ENCRYPTION_KEY` (старый шифртекст просто перешифровывался бы заново, оригинал терялся).

Теперь: plaintext детектится по формату реального Telegram-токена — регексп `^\d+:` (цифры, двоеточие, тело). Только такие строки шифруются. Если строка не подходит под этот формат И не расшифровывается текущим ключом — она не трогается, в лог уходит `warning` с id бота. Fernet-шифртекст физически не может совпасть с этим regex (он всегда начинается с `g` в base64, никогда с цифры) — коллизий нет. Протестировано: plaintext → шифруется; уже зашифрованное текущим ключом → не трогается; «мусор»/шифртекст под другим ключом → не трогается, только warning.

### Бонус — `from __future__ import annotations`
Добавлена в `services/claude_service.py` и `templates/trip_manager.py` (единственные два файла в проекте, где отсутствовала при использовании `X | None` в аннотациях). Без неё эти файлы падают при импорте на Python 3.9 (нет проблем на 3.10+, но лучше не зависеть от версии интерпретатора). Остальные файлы (`accountant.py`, `booking_beauty.py`, `manager_secretary.py`) уже её содержали — несогласованности не было. Проверено: `services/claude_service.py` теперь импортируется и на Python 3.9, и на 3.12 без ошибок. Ревью подтвердило безопасность фикса — в проекте нигде нет рантайм-интроспекции аннотаций (`inspect.signature`/`typing.get_type_hints`/pydantic), так что `from __future__ import annotations` не может ничего сломать в поведении.

### Задача 3 — systemd
Добавлен `deploy/bot-creter.service`:
```ini
[Unit]
Description=Bot Creter — Telegram bot factory
After=network-online.target
Wants=network-online.target
StartLimitIntervalSec=60
StartLimitBurst=5

[Service]
Type=simple
User=REPLACE_WITH_SERVER_USER

```

**Перед запуском на сервере замени плейсхолдеры:**
- `REPLACE_WITH_SERVER_USER` → системный пользователь, от которого крутится фабрика
- `/REPLACE/PATH/TO/bot-creter` → реальный путь к репозиторию на VDS (3 места)

**Порядок деплоя на сервере:**
```bash
cd /path/to/bot-creter
git pull origin master
source venv/bin/activate && pip install -r requirements.txt
sudo cp deploy/bot-creter.service /etc/systemd/system/bot-creter.service
sudo systemctl daemon-reload
sudo systemctl enable bot-creter        # автозапуск при загрузке — один раз
sudo systemctl restart bot-creter       # применить изменения
sudo systemctl status bot-creter        # проверить, что поднялся
journalctl -u bot-creter -f             # смотреть логи
```
`StartLimitIntervalSec=60` / `StartLimitBurst=5` — если фабрика падает 5 раз за 60 секунд (например, из-за неверного `ENCRYPTION_KEY`), systemd остановит попытки рестарта вместо бесконечного краш-лупа. Починить причину и выполнить `sudo systemctl reset-failed bot-creter && sudo systemctl restart bot-creter`.

### Задача 5 — шифрование токенов
- Токен шифруется перед `INSERT` (в Раунде 3 функция переименована в `create_bot_record_with_admins()`, см. Задачу 7 Раунда 3) и расшифровывается прозрачно в `get_all_bots()` / `get_bot()` / `get_bot_by_name()` — весь остальной код (main.py, handlers/*, services/bot_runner.py) как получал/передавал plaintext-токен, так и продолжает, ничего в них менять не пришлось.
- Алгоритм: `cryptography.fernet.Fernet`, ключ — `ENCRYPTION_KEY` из `.env`.
- Разовая миграция (`migrate_encrypt_tokens()`) вызывается автоматически при каждом `init_db()`: сканирует все токены, plaintext (не расшифровывается текущим ключом) — шифрует и сохраняет; уже зашифрованные — не трогает. Идемпотентна, безопасна при каждом рестарте.
- При отсутствии или невалидности `ENCRYPTION_KEY` — модуль падает при импорте с понятным сообщением (аналогично существующей проверке `BOT_TOKEN`/`ANTHROPIC_API_KEY`), а не тихо работает без шифрования.
- Токены нигде не логировались в открытом виде — проверено по всему проекту, менять не пришлось.

**⚠️ Известное ограничение (сознательно оставлено вне рамок Stage 1):** если `ENCRYPTION_KEY` сменить после того, как токены уже зашифрованы, старые токены станут нечитаемыми — расшифровка потерпит неудачу, и **при следующем рестарте они будут молча зашифрованы заново новым ключом** (значение попадёт в лог `Could not decrypt a bot token` на уровне `warning`, но сам токен восстановить будет нельзя). Это не бага, а следствие простой схемы без версионирования шифртекста — умышленно, чтобы не усложнять Stage 1. **Правило: `ENCRYPTION_KEY`, заданный один раз, менять нельзя.** Если ключ утерян/скомпрометирован — потребуется вручную пересоздать токены ботов через BotFather и обновить их в БД.

## Как проверить после деплоя

1. **Персистентность:** `sudo systemctl restart bot-creter` (или перезагрузка сервера целиком) → зайти в `DATA_DIR` и убедиться, что `bots.db`, `generated_bots/*.py`, `bot_images/`, `bot_avatars/` на месте; в логе (`journalctl -u bot-creter`) увидеть строку `restore_bots: N restored, 0 failed` с N > 0, если до рестарта были запущенные боты.
2. **Шифрование:** открыть `bots.db` любым sqlite-клиентом (`sqlite3 data/bots.db "SELECT token FROM bots LIMIT 1;"`) — значение должно начинаться с `gAAAAA...` (Fernet), а не быть похожим на `123456:AAH...` (формат реального Telegram-токена).
